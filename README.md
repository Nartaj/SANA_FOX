# 🦊 Sana Fox: Интерактивная платформа обучения кибербезопасности

## 🏛 Обзор Архитектуры (Architecture Overview)

Проект спроектирован на основе строгого паттерна **Слоистой Архитектуры (Layered Architecture)**. Этот подход обеспечивает высокую масштабируемость, четкое разделение ответственности (Separation of Concerns) и абстракцию бизнес-логики от UI-представления. Архитектура приложения строится на следующих уровнях:

1. **Data Layer (Слой данных)**: Инкапсуляция персистентности (надежное хранение состояния) и статических баз знаний.
2. **Service Layer (Сервисный слой)**: Изоляция сложной бизнес-логики, валидация процессов и оркестрация независимых доменных правил.
3. **Logic Layer (Слой управления)**: Игровые контроллеры и пользовательские хуки, связывающие потоки данных и слой представления.
4. **View Layer (Слой представления)**: Функциональные UI-модули, анимации и интерактивные экраны приложения.

## ⚙️ Корневая конфигурация (Root Configuration)

Корневые файлы образуют технологический фундамент проекта, критически важный для типобезопасности, оптимизации сборки и контроля зависимостей:

* **`vite.config.ts`**: Оптимизированная конфигурация сборщика Vite. Отвечает за обеспечение мгновенного HMR (Hot Module Replacement) в контексте локальной разработки и высокопроизводительную компиляцию статических ассетов для production-среды.
* **`tsconfig.json`**: Строгая настройка компилятора TypeScript (Strict Mode). Гарантирует абсолютную **Type-Safety** (типобезопасность) на всех архитектурных уровнях. Минимизирует риск runtime-исключений за счет статического анализа при высоконагруженных операционных сопоставлениях (например, интеграция логики ID игровых карт).
* **`package.json`**: Инфраструктурный манифест проекта. Управляет версионированием ключевых пакетов (React 19, Tailwind CSS v4, Framer Motion) и декларирует детерминированные скрипты экосистемы конвейера интеграции (`dev`, `build`, `lint`).

## 📂 Глубокий разбор структуры директорий (Deep-Dive Directory Breakdown)

```text
/sana_fox_app
├── 📁 src
│   ├── 📁 data           # Persistence Layer (Слой персистентности)
│   ├── 📁 hooks          # Logic Layer (Слой контроллеров/Игровой движок)
│   ├── 📁 services       # Business Logic Layer (Управление правилами)
│   ├── 📁 types          # Domain Model Layer (Доменные модели структуры)
│   ├── 📁 navigation     # Routing & State (Модуль навигации)
│   └── 📁 screens        # View Layer (UI Модули/Представление)
├── 📄 package.json       # Описание зависимостей
├── 📄 tsconfig.json      # Типобезопасность ядра
└── 📄 vite.config.ts     # Конвейер сборки
```

### 1. `src/data` (Persistence Layer)
Фундаментальный слой, отвечающий за долговременное хранение пользовательских сессий и статичные структуры данных.
* **`repository.ts`**: Бесшовно реализует паттерн **DAO (Data Access Object)** для работы с `LocalStorage`. Строго абстрагирует низкоуровневые CRUD-операции над профилем пользователя, инкапсулируя все особенности реализации браузерного хранилища.
* **`cards.ts`**: Является статической **Knowledge Base (Базой знаний)** для Card-Battle системы. Декларирует иммутабельные объекты карт (типы "Attack" и "Defense"), описывает их семантические связи, математические показатели урона и уровень редкости.

### 2. `src/hooks` (Logic Layer)
Слой оркестрации игровых механик, связывающий сложные сервисные операции с реактивным деревом React.
* **`useCyberGame.ts`**: Ключевой компонент архитектуры — **Game Engine Controller**. Выступает в роли конечного автомата переходов состояний. Анализирует парные столкновения (Attack/Defense matching), вычисляет наносимый урон через паттерны уязвимостей и возвращает окончательный детерминированный исход раунда для последующей трансформации в UI.

### 3. `src/services` (Business Logic Layer)
Инкапсулирует сложную доменную логику и независимые от представления глобальные бизнес-правила (Business Rules).
* **`AchievementService.ts`**: Ядро подсистемы геймификации. Асинхронно анализирует потоки данных из сессий (например, прохождение квиза без урона) и запускает защищенные триггеры разблокировки уникальных достижений.
* **`DailyChallengeService.ts`**: Контроллер циклических квестов. Управляет псевдослучайной генерацией задач из пула карт, отслеживает кулдауны 24-часовых интервалов и отвечает за консистентную валидацию бонусного наградного опыта (XP).

### 4. `src/types` (Domain Model Layer)
Основа объектно-ориентированной модели и безопасности типов в приложении. Включает строгие инфраструктурные TypeScript-интерфейсы:
* **`Card`**: Формальный контракт игровой карты (включая уникальный идентификатор, параметр защиты/атаки и семантический атрибут перекрытия `beats`).
* **`UserProgress`**: Модель динамического состояния индивидуального игрока (капитализация глобального уровня, метрика XP, длительность актуальных стриков).
* **`BattleResult`**: DTO-модель результата вычисления боевого хода (фактически нанесенный урон, булевый индикатор победы/поражения, текстовый образовательный фидбек).

### 5. `src/navigation` (Routing System)
Архитектурный узел маршрутизации, обеспечивающий гладкий опыт **Single-Page Application (SPA)** без деструктивных перезагрузок браузера.
* **`NavigationState.tsx`**: State-driven контроллер потока. Сохраняет историю переходов в контексте компонента (Context API) и безупречно интегрируется с паттерном **Bottom Navbar**, обеспечивая Mobile-First пользовательский контроль.

### 6. `src/screens` (View Layer)
Коллекция функционально изолированных модулей представления, отвечающих за конкретные ветки пользовательского сценария:
* **Arena (`ChallengesScreens`, `CheckScreens`)**: Высокоинтерактивная среда для практической проверки знаний через динамическую механику Card-Battle.
* **Academy (`LessonsScreens`)**: Структурированный теоретический модуль для поэтапного прохождения учебной базы векторов кибератак.
* **Profile (`ProfileScreens`, `AchievementsScreens`)**: Среда аналитики и персонализации для просмотра детальной статистики и коллекционных вознаграждений.

## 🔄 Поток выполнения (Logic Flow)

Проект непреклонно следует архитектурному принципу однонаправленного потока данных (Unidirectional Data Flow), что гарантирует высочайшую предсказуемость жизненного цикла:

1. 👉 **User Action (Действие)**: Пользователь выбирает карту противодействия (Defense) на странице Arena (Слой `View`).
2. ⚙️ **Hook Logic (Вычисление)**: `useCyberGame.ts` (`Logic Layer`) перехватывает событие, обрабатывая характеристики карты через сопоставление параметра `Attack ID` с атрибутом защиты `beats`.
3. 🛡️ **Service Validation (Валидация)**: Аналитические сервисы проверяют критерии успешности операции: инкрементируют счетчики стриков (`DailyChallengeService`) и сканируют доступность новых ачивок (`AchievementService`).
4. 💾 **State Update (Персистентность)**: Объект `repository.ts` (`Data Layer`) атомарно записывает обновленные мутации состояния (XP, урон, прогресс) в хранилище.
5. 🖥️ **View Render (Отрисовка)**: React перехватывает изменение глобального контекста и реактивно перерисовывает UI-дерево, транслируя анимацию отраженной атаки и выводя детальный обучающий совет от системы.

## 🛠 Технологический стек (Tech Stack)

| Компонент / Библиотека | Роль в проекте | Архитектурное предназначение |
| :--- | :--- | :--- |
| **React 19** | View Layer (UI-ядро) | Декларативный рендеринг, компонентная композиция. |
| **TypeScript** | Доменно-ориентированный ЯП | Гарантия типобезопасности бизнес-логики и моделей на этапе компиляции. |
| **Vite** | Сборщик (Bundler) | Максимальная скорость холодного старта, HMR и оптимизация графа зависимостей дерева модулей. |
| **Tailwind CSS v4** | Стилизация (Utility-First) | Независимый дизайн-системный подход, гибкая адаптивность, нулевой runtime-overhead. |
| **Framer Motion** | Движок анимаций | Рендеринг плавных микроинтеракций для 60 FPS Card-Battle механики. |
| **Gemini Flash API** | Динамическая генерация | Адаптивная вариабельность игровых сценариев, контекстно-зависимый Feedback и генерация угроз на лету. |
